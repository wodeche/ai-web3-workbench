# 交易解释器 — 概念设计

> 实践目标：理解 LLM 输出的三层分类（事实 / 推断 / 不确定），学会何时信模型，何时必须验证。

## 核心问题

给 LLM 一个交易 hash（如 `0xabc123...`），让它解释这笔交易在做什么。LLM **无法直接查询区块链**，所以它的回答可能是：

- 🟢 **链上事实**：可验证的真理（需外部数据源提供）
- 🟡 **模型推断**：基于训练数据的模式识别，可能对可能错
- 🔴 **不确定 / 幻觉**：模型猜的、编的，没有依据

## 设计思路

交易解释器不是让 LLM 直接「猜」交易内容，而是设计一个流程：

```
                   ┌─────────────────┐
  tx hash ────────▶│  数据获取层      │
                   │  (Etherscan API  │
                   │   / RPC node)   │
                   └────────┬────────┘
                            │ 链上数据 (JSON)
                            ▼
                   ┌─────────────────┐
                   │  LLM 分析层      │
                   │  prompt 要求：   │
                   │  分层输出        │
                   └────────┬────────┘
                            │ 结构化输出
                            ▼
                   ┌─────────────────┐
                   │  输出            │
                   │  - 链上事实      │
                   │  - 模型推断      │
                   │  - 不确定项      │
                   └─────────────────┘
```

**关键原则**：LLM 不查链，只解释数据。数据由工具提供。

## 输出结构设计

```json
{
  "tx_hash": "0x...",
  "on_chain_facts": {
    "from": "0x...",
    "to": "0x...",
    "value": "1.5 ETH",
    "gas_used": "21000",
    "method": "transfer(address,uint256)",
    "status": "success",
    "block_number": 12345678
  },
  "model_inference": {
    "human_readable": "这是一笔向 0x... 转账 1.5 ETH 的普通交易",
    "likely_purpose": "个人转账 / 交易所提现",
    "risk_signals": [],
    "confidence": "high",
    "reasoning": "标准 transfer() 调用，gas 正常，无异常数据"
  },
  "uncertainties": [
    {
      "question": "接收地址是否属于交易所？",
      "why_uncertain": "需要链下标签数据库确认"
    }
  ],
  "requires_verification": false
}
```

## 三层分类规则

| 层级 | 定义 | 来源 | 例子 |
|------|------|------|------|
| **链上事实** | 从链上数据直接读取，不可争议 | Etherscan / RPC 返回的字段 | from, to, value, gas, status |
| **模型推断** | 基于事实的推理，有依据但需标明置信度 | LLM 分析 + 标注 reasoning | "这笔交易像是套利"（标注：基于方法签名和路径判断） |
| **不确定** | 模型无法判断、缺少信息、或可能编造的 | 显式声明 | "对方地址是什么实体？无法从链上判断" |

## 为什么这个练习重要？

1. **Hallucination 实战理解**：如果直接让 LLM 解释 tx hash，它会编造交易内容
2. **Agent 设计基础**：未来的 AI Agent 需要「工具调用 → 数据 → LLM 分析」的流水线
3. **Web3 特有问题**：链上数据是公开的，但解释它需要 domain knowledge

## 扩展方向

- 接入 Etherscan API 获取真实交易数据
- 增加风险检测规则（无限授权、黑名单地址）
- 用结构化输出（JSON Schema）约束 LLM 格式
